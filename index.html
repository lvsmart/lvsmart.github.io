accessToken

  • 使用企业微信的API，必须带有accessToken
  • accessToken由appid和secret生成，生成之后有效期2小时，每天调用次数受限

简单方案

  • 不能每次调用接口，都生成accessToken。
  • 生成的accessToken进行缓存，token过期了，再去生成

问题

  • 当有多个请求token同时过期，就会同时触发多个refresh
  • 返回多个token，但只有最后一个token是可用的
  • 其他请求token认证再次失败，再次同时触发多个refresh

解决方案

  • 使用redis缓存token，定时刷新token
  • 请求如果token过期，直接报错，不再触发刷新机制
  • 将多节点刷新token，变为单节点刷新token

引入的新问题

  • 需要维护redis和定时任务
  • 如果修改了appid或secret，需要主动触发token刷新

改进的技术

  • django + celery + redis，使用celery管理定时任务
  • django + action，使用django后台，在修改appid或secret之后，主动触发刷新

python连接mysql

背景

  • python3已经成为主流，python2基本不再使用。
  • 无论是开发环境还是生产环境，mysql都是不二选择。
  • mysql版本一直保持稳定更新，性能和功能也在不断加强。

Driver

  1. mysqlclient
    • 速度快，使用C，安装有依赖
    • 生产环境下推荐使用
  2. PyMySQL
    • 纯python实现，安装使用简单
    • 当安装mysqlclient失败的时候使用

ORM

  1. SQLAlchemy
    • ORM的第一选择
    • 对于各种DB的各种Driver都有很好的支持
  2. peewee
    • 非常轻量，API类似Django ORM
    • 如果原有技术栈基于Django，可以使用。

dubbo错误跟踪

发表于 2019-03-08 | 更新于 2019-10-13 | 分类于 运维 | 评论数： | 阅读次数：

起因

  • 发版的时候，应用报错

简单排查

  • 在停止了dubbo之后，立刻停止service，错误随之而来。
  • 两种可能：1、dubbo没有停掉；2、停止service时，应用自身错误

其他信息

  • 两台同样的主机，同时执行，只有一台出错。
  • 四个主机组成的集群，一次发版过程中，只有一台出错。
  • 发版系统只发生过启动dubbo失败，没有发生过停止dubbo失败。

之前类似的错误

  • 因为dubbo名称和应用名称不一致，导致停止dubbo的时候，没有找到dubbo，导致没有停dubbo，直接停应用。
  • 此种情况下，如果操作的是服务提供者应用A，那么，报错的应该是服务消费者B，C等
  • 而此次报错，停止的是服务提供者应用A，报错的依旧是服务提供者应用A。

查错

分析ELK日志，主机日志，查阅具体报错的java代码，询问具体开发人员

分析ELK日志

  • 根据ELK的报错时间，结合发版系统的操作时间，发现跟停dubbo和停service有直接关系。
  • 报错集中在同一个方法的同一行，初步怀疑是一个批量任务的for循环出错
  • 错误：dubbo服务找不到。但是，是其他服务提供者的接口，即，此时应用作为一个消费者报错。

主机日志

  • 因为发版时候的备份策略，以及应用日志的打印方式，导致主机日志在查错过程中，并没有起到应有作用。

具体java代码

  • LTS定时任务，每三天的晚上1点执行。

开发人员

  • 定时任务出错，之后任务会再次执行，没有影响
  • 此定时任务执行需要很久时间，期间停service，自然会出错。

总结

  • 运维可以根据ELK错误日志，以及自身的操作日志，发现并排查出问题的50%以上内容
  • 错误可以类比，但具体到个例，就必须看具体的错误，如果能看懂java代码，可以具体到那个功能出错。
  • 引发错误的原因很多，甚至有可能是共同作用。
  • 应该先把所有的原因都罗列出来，而不能过早排除某种错误。

windows搭建类unix开发环境

发表于 2019-01-30 | 更新于 2019-10-13 | 分类于 运维 | 评论数： | 阅读次数：

背景介绍

  • 运维开发的项目很多，其中有部分python和go的项目，依赖的类库不支持windows。（如kafka，ansible）
  • 日常工作依赖的企业微信，WPS办公软件，keepass等，离不开windows。
  • 综上，需要在windows系统中，打造类unix开发环境

系统要求

  • windows 10 最新版本
  • 开启WSL功能，并安装ubuntu，启动，设置用户名密码

使用软件

  • mobaxterm，免费版本即可

具体步骤

1、准备工作

  • mobaxterm会自动识别WSL安装的ubuntu,，创建一个默认配置的WSL session即可。
  • 使用国内ubuntu源，加速接下来的软件安装。常用的有：清华，阿里云，网易，中科大
  • 清华大学镜像站

2、安装vscode

  • 在官网下载ubuntu版最新deb包

  • 此时，窗口不能移动，不能全屏，还存在中文字体，以及中文输入法问题。（并非乱码，而是fonts问题）

  • 但是已经能够安装插件，代码高亮，跳转等正常开发功能了

3、安装xfce4

  • wsl默认使用ubuntu-server，没有桌面，安装一个轻量级桌面xfce4即可

  • 然后再新建一个图形化的session，具体配置如下图

  • 然后安装配置输入法（推荐fcitx + google拼音输入法）

4、开发环境

  • go直接使用源码

  • python系统自带python3（python3.6）

  • 一般需要额外安装gcc， python3-venv，python3-dev等一系列开发依赖包

5、最终效果

  • 项目代码可以直接使用windows下原有代码，配置好go path， venv等环境即可

travis部署page

发表于 2019-01-15 | 更新于 2019-10-13 | 分类于 运维 | 评论数： | 阅读次数：

个人博客简介

  • 使用hexo框架：先用markdown编写，然后渲染成html—+css+js的静态网站
  • source目录下，是markdown文件，可以认为是程序源码。
  • public目录下，是编译好的网站，可以看做可执行程序。

使用travis的背景

  • 使用hexo引擎+next主题搭建的个人博客静态网站，本身可以很方便的deploy到github的page页面。但是不方便国内访问，所以需要在国内vps部署一份。
  • 写blog可能在各种环境：家里，公司，或者直接在vps上面。此时，markdown文件的同步以及版本管理，就成了问题。
  • 综上，在github上面创建两个项目：
    • page项目存放生成的html等静态文件，可以直接访问。
    • 源文件项目存放hexo创建的目录，包括所有的source，theme，image等，方便在各个环境下编写blog
  • 使用travis自动部署

使用travis方法

  • 保存markdown源文件
  • 自动build push
